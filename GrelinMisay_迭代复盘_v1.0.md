## 四、开发过程中发现并修复的问题

### 4.1 Bug修复记录

| # | 问题描述 | 严重级别 | 修复方案 | 影响范围 |
|---|---------|---------|---------|---------|
| 1 | `g_config.py` 不是FastAPI路由器，导致 `/api/config/actions` 返回404 | 高 | 删除 `app/api/g_config.py`（仅为常量定义文件，从未在router.py注册） | 训练动作数据获取 |
| 2 | 前端训练动作数据硬编码在组件中，不经过后端API | 中 | 改为 `useEffect` + `trainingAPI.actions()` 动态获取，失败回退到 `DEFAULT_ACTIONS` | training/index.tsx + api.ts |
| 3 | GitHub推送失败（443端口连接超时） | 低 | 沙箱环境网络限制，需用户本地手动执行 `git push` | 代码发布 |
| 4 | 新增 `login_by_code` 端点返回404 | 中 | 后端未重启导致旧进程占用端口，`taskkill` + 重新 `python run.py` 解决 | 验证码登录 |

| Token存储在内存字典，重启失效 | Phase 4迁移JWT+Redis |

## 五、技术实现质量评估

### 5.1 架构评分

| 维度 | 评分 | 说明 |
|------|------|------|
| **代码结构** | 8/10 | 前后端分离清晰，API模块化，但部分后端路由函数臃肿 |
| **数据安全** | 7/10 | SHA256密码哈希，Token鉴权覆盖全API，但Token内存存储不持久 |
| **错误处理** | 6/10 | 后端有try-catch和HTTP异常，前端仅有console.log |
| **代码复用** | 7/10 | request()统一封装，Taro适配层设计合理 |
| **可维护性** | 7/10 | 模块职责清晰，路由注册集中，schema定义完整 |
| **可扩展性** | 8/10 | 模型预留了Phase 2/3字段（goal_type, members, description等） |

### 5.2 各模块深度评价

#### 账号系统 (g_auth)
- **亮点**：注册/登录接口返回统一APIResponse格式，Token中间件可复用；支持密码+验证码双模式登录，验证码有过期机制（5分钟）
- **不足**：Token无过期自动清理，验证码仍为固定值模拟

#### 目标管理 (g_goals)
- **亮点**：打卡进度自动计算（+10%→100%→completed），用户数据隔离
- **不足**：缺少打卡日历记录表（未使用g_goal_checkins），进度只增不减

#### 训练记录 (g_training)
- **亮点**：多组训练参数完整（次数/重量/休息/感受），selectinload预加载优化
- **不足**：无训练模板保存/使用功能（PRD列出的需求）

#### 日历 (g_calendar)
- **亮点**：聚合查询设计（Goal+CalendarEvent合并），颜色语义化
- **不足**：缺少手动创建、编辑、删除事件的API

#### AI助手 (g_ai)
- **亮点**：规则引擎+LLM双通道降级设计，消息持久化
- **不足**：规则引擎覆盖场景有限（6种），无对话上下文记忆

## 九、团队效能与流程反思

### 9.1 做得好的

- **架构设计合理**：前后端分离+API模块化，代码结构清晰，后续扩展成本低
- **数据库模型前瞻性**：8张业务表覆盖了Phase 1-3的大部分需求字段
- **统一响应格式**：`APIResponse` 封装使前端处理标准化
- **文档完善**：PRD + README + DEPLOY_GUIDE 三文档齐全
- **降级设计**：AI模块规则引擎+LLM双通道；前端动作库API失败回退默认数据

### 9.2 需要改进的

- **前端编辑功能滞后**：3个后端API已实现但前端未暴露UI（目标编辑/个人资料编辑/目标统计）
- **错误提示不够友好**：前端catch仅console.log，用户看不到具体错误原因
- **样式分散**：CSS未使用CSS变量统一管理主题色
- **缺少单元测试**：后端和前端均无自动化测试覆盖
- **Git提交粒度**：MVP全部内容在单一commit，不利于代码审查和回滚

### 9.3 下一迭代改进措施

1. **拆分Git提交**：每个功能模块独立commit
2. **补充前端编辑UI**：优先补齐目标编辑和个人资料编辑
3. **添加错误提示**：统一Toast错误提示，区分网络错误/业务错误
4. **CSS变量化**：将PRD定义的颜色体系抽为CSS变量
5. **编写核心测试**：至少覆盖认证和目标打卡的关键路径
